# UI图集的一些优化项

# 精灵图集

UI也算是性能的重灾区了，GPU、CPU及内存都有涉及，其中很重要的一部分就是精灵图集，这里将会同时涉及到这三个方面

## 方案讨论：动态图集

动态图集是现在使用的一个可选项，也就是将零散的图片在运行时生成指定的图集

优点自然是可以灵活的构造新的图集，运用得当可以**最大程度的合批**；但是缺点也很明显，一方面是会**对内存造成压力，**主要体现在动态合批的资源需要开启read/write，导致双倍内存占用，另一方面就是动态合批的这个过程，等于是将开发时的生成图集任务转移到了运行时，也即是将会**对CPU造成压力**。

|模块||
|---|---|
|CPU|将开发期的任务推迟到运行时，给**运行时CPU增加压力**|
|GPU|**合理构造、且资源合并后大小不超过2048的前提下**：当前所有ui都是合批的，也即是1个批次。**且预定义生成后的图集最小为2048**|
|内存|资源**占用双倍内存**|

## 方案讨论：静态图集

静态图集是目前市面上大多数项目使用的一个方式，通过将一个模块/一组资源打成一个图集，从而实现策略上的最大程度合批，且打图集的操作都是在开发期进行的。因此这个方案比较稳定与均衡。

|模块||
|---|---|
|CPU|图集的生成都是开发期进行的|
|GPU|图集分配合理的情况下，只有很少的批次(如界面一个dc，通用图集一两个dc，通用图标一两个dc)|
|内存|资源单份内存|

## 更多讨论：鱼与熊掌兼得

无论是以上的哪种方案，无非是空间、时间，以及不同时期的时间进行的博弈。没有最完美的方案，只有最适合的方案。

静态图集是Unity默认提供的方案，也是最吃得开的方案。动态图集也有比较合适的方案，感兴趣的可以一起讨论尝试下

## 方案执行

### 采用Unity图集方案

由于现在的动态图集有比较大的限制，精灵图在内存中也有着很大的占比了，所以双倍内存是很伤的一件事，一直以来也没有启用，所以目前比较合适的方案是使用Unity的图集方案，可以最小改动最快见效，性价比最高。

### 执行效果

目前执行下来**dc减少50%以上！**

### 持续优化

有以下几个方面可以持续优化：

1. UI静动分离，图文分离，稳定合批对象的合批条件
2. ~~规范资源，将更多UI 2.0目录下的散图迁移到Texture/Atlas目录下~~
3. 不使用图集外的资源。尤其是UGUI默认资源，即使是一个小方块
4. 图集的策略需要调整，**尽量一个界面或者一个模块单独一个文件夹**，而非固定的几个文件夹一直塞资源

## 未来方向

关于精灵图集的方案，未来主要有两个可选方案供讨论和研究，以便落实到执行

### 方向一：坚持静态图集方案

静态图集本身是个均衡的方案，只要合理配置资源，规范使用资源，就可以很方便的达到一个很好的效果。未来将会抽象一个独立的图集模块，以此**实现图集资源的动态加载、卸载，保证内存使用合理**。

### 方向二：转为动态图集方案

这方面主要是考虑项目情况分析痛点，如：资源双份内存问题、运行时生成的效率、兼容性、手动合批等问题能否解决，能解决几个，以及使用此方案的性价比。

# **开发流程**

## 需要代码动态加载的资源

这部分资源统一放在`Assets/Textures/Atlas`下， 一个文件夹一个类型， 创建(或复制)一个新的`.spriteatlas`文件,在`Asset/Textures/Atlas/AtlasPacker`下，将文件夹添加到`.spriteatlas`的引用中。

同时配置`GlobalAtlas.lua`中的`textureType、textureConfig`和定义的读取函数。

## 引用到的静态加载资源

这部分资源统一放在UI2.0目录下，一个模块/界面一个单独的文件夹。

其他的无需任何处理，会**自动进行图集处理**，请注意git自动修改判断是否需要提交。

## 需要注意的内容

UI2.0下会存在不同格式的资源，而图集是只针对Sprite的，所以**效果图请不要转为Sprite2D格式**。如果需要叠在预制体下面制作，可以使用RawImage组件。

另外，比较大的资源(大于512)可以直接使用Texture2D,也不要转为Sprite，**否则将会使图集的利用率变低**。

可以在以下目录预览图集情况：

- `Assets/Textures/Atlas/SpriteAtlas`
- `Assets/Textures/UI2.0/SpriteAtlas`
