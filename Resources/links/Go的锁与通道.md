#Go

| 方法           | 适用场景       | 注意事项                   |
|--------------|------------|------------------------|
| 互斥锁（Mutex）   | 简单的共享变量保护  | 避免锁嵌套，防止死锁             |
| 通道（Channel）  | 协程间数据传递和同步 | 明确关闭通道，避免阻塞            |
| 原子操作（Atomic） | 简单类型的原子操作  | 仅支持基本类型（int32, int64等） |
| 读写锁（RWMutex） | 读多写少场景     | 写锁优先级高于读锁              |
| Sync.Map     | 高频并发读写     | 无需手动锁，但性能优化需实测         |

| 场景            | 是否需要关闭通道   | 说明                            |
|---------------|------------|-------------------------------|
| 通道是“单向发送”的生产者 | 必须关闭       | 否则消费者协程无法知道何时结束               |
| 通道是“临时通信”的工具  | 无需关闭       | 例如仅用于协程间触发信号（如 chan struct{}） |
| 通道作为参数传递      | 由发送方决定是否关闭 | 遵循“谁创建谁关闭”或明确约定               |

| 场景          | 是否关闭通道 | 原则             |
|-------------|--------|----------------|
| 临时通信（一次性信号） | 无需关闭   | 通过数据发送完成隐式结束   |
| 生产者-消费者模式   | 生产者关闭  | 遵循“谁创建谁关闭”原则   |
| 多协程共享通道     | 由主协程关闭 | 确保所有协程完成任务后关闭  |
| 参数传递的只读通道   | 不可关闭   | 接收方无关闭权限（编译保护） |

#### **不关闭通道会导致协程泄漏吗？**

- **取决于接收方逻辑**：
    - 若接收方使用 `for range` 循环，**必须关闭通道**，否则协程永久阻塞。
    - 若接收方使用 `select` 或有限次数的读取，可不关闭（需确保逻辑正确）。

#### **Q：如何避免关闭已关闭的通道？**

- **方案**：通过 `sync.Once` 或状态标记确保通道只关闭一次。



```go
var closeOnce sync.Once
func safeClose(ch chan int) {
    closeOnce.Do(func() {
        close(ch)
    })
}
```
