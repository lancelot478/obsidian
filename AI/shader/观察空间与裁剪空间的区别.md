# 观察空间与裁剪空间的区别

> 日期：2026-03-31

---

## 坐标空间变换流水线

顶点在 GPU 里要经过一条固定的变换流水线：

```
模型空间          观察空间          裁剪空间          NDC空间          屏幕空间
(Object Space) → (View Space) → (Clip Space) → (NDC Space) → (Screen Space)
       ↑                ↑               ↑              ↑
    Model矩阵       View矩阵        Proj矩阵        透视除法(自动)
```

每个空间都有自己的"参照系"，理解它们的本质差异是写好 Shader 的基础。

---

## 各空间定义与特征

### 模型空间（Object Space）
- **原点**：模型自身的中心（美术建模时定义）
- **特征**：坐标与模型旋转/位移无关，一把枪的枪口永远在 (0, 1, 0)
- **Unity矩阵**：`UNITY_MATRIX_M`（Model）

### 观察空间（View Space / Camera Space / Eye Space）
- **原点**：摄像机位置
- **坐标轴**：X=右，Y=上，**Z=摄像机朝向的反方向（OpenGL/Unity）**
- **特征**：所有坐标都是"从摄像机看过去"的相对位置，物体离摄像机越远，Z 值越大
- **Unity矩阵**：`UNITY_MATRIX_V`（View）；模型→观察：`UNITY_MATRIX_MV`

### 裁剪空间（Clip Space / Homogeneous Clip Space）
- **原点**：屏幕中心
- **特征**：投影矩阵已作用，但**尚未做透视除法**；坐标带有 W 分量（齐次坐标）；GPU 用这个空间做视锥裁剪
- **范围**：`-W ≤ x,y,z ≤ W`（W 不等于 1）
- **Unity矩阵**：`UNITY_MATRIX_P`（Projection）；观察→裁剪：`UNITY_MATRIX_VP`

### NDC 空间（Normalized Device Coordinates）
- **原点**：屏幕中心
- **特征**：GPU 自动完成透视除法（÷W）后得到，坐标范围归一化到 `[-1, 1]`
- **这步不需要手动写**，GPU 自动完成

### 屏幕空间（Screen Space）
- **原点**：屏幕左下角（或左上角，取决于平台）
- **特征**：最终的像素坐标，单位是像素

---

## 关键区别对比

| 对比项 | 观察空间（View Space） | 裁剪空间（Clip Space） |
|------|-------------------|-------------------|
| 变换阶段 | 乘完 View 矩阵后 | 再乘完 Projection 矩阵后 |
| 是否投影 | ❌ 没有，仍是三维线性空间 | ✅ 已投影，空间被"压扁" |
| W 分量 | 通常 = 1（线性） | W ≠ 1（携带深度信息） |
| 距离含义 | Z 轴直接代表真实深度距离 | Z/W 才是深度（非线性） |
| 空间形状 | 梯形视锥（真实三维） | 立方体（投影后归一） |
| 适合做什么 | 几何计算、法线变换、光照 | 顶点最终输出、裁剪判断 |

---

## 直觉理解

```
观察空间 = 你站在摄像机位置，用尺子量出的真实三维世界
           物体在你左边 3 米，上方 2 米，前方 5 米

裁剪空间 = 摄像机已经拍下照片，照片上的坐标
           但照片还没洗出来（尚未 ÷W），坐标带有透视"压缩"
```

---

## 什么情况下用哪个空间？

### 用观察空间的场景

**① 描边宽度均匀（上一个 Shader 的核心原因）**

```hlsl
// ✅ 观察空间膨胀：宽度与距离无关，视觉均匀
float4 viewPos    = mul(UNITY_MATRIX_MV, v.vertex);
float3 viewNormal = mul((float3x3)UNITY_MATRIX_IT_MV, v.normal);
viewNormal.z = 0;
viewPos.xy  += viewNormal.xy * _OutlineWidth;
o.pos        = mul(UNITY_MATRIX_P, viewPos);

// ❌ 模型空间膨胀：近大远小，越远轮廓越粗
o.pos = UnityObjectToClipPos(v.vertex + v.normal * _OutlineWidth);
```

**② 屏幕空间环境光遮蔽（SSAO）**

SSAO 需要采样当前像素周围的真实深度，观察空间的 Z 是线性深度，计算距离准确。裁剪空间的深度是非线性的，无法直接做距离计算。

**③ 光照计算（Blinn-Phong 等）**

大量光照模型在观察空间计算，因为"摄像机方向"就是 (0,0,-1)，不需要额外传入摄像机位置，节省计算量。

```hlsl
// 观察空间中，摄像机方向是常数，不需要额外传参
float3 viewDir = float3(0, 0, -1); // 摄像机就在原点，看向 -Z
```

**④ 粒子拉伸、Billboard 对齐**

粒子始终面向摄像机，在观察空间里只需把 XY 轴固定即可，不受模型旋转影响。

---

### 用裁剪空间的场景

**① 顶点着色器最终输出（必须）**

```hlsl
// 顶点着色器的输出 SV_POSITION 必须是裁剪空间坐标
// 这是 GPU 硬件规定的，没有例外
o.pos = UnityObjectToClipPos(v.vertex); // 一步到位：模型→裁剪
```

**② 屏幕空间特效（获取屏幕 UV）**

```hlsl
// 在裁剪空间坐标上做透视除法，得到屏幕 UV
// ComputeScreenPos 就是在裁剪空间做这件事
float4 screenPos = ComputeScreenPos(o.pos);
// 之后在片元着色器 screenPos.xy / screenPos.w 得到屏幕UV
```

**③ 深度偏移（Depth Bias）**

直接在裁剪空间修改 Z，可以让物体在深度上偏移，用于解决 Z-Fighting：

```hlsl
o.pos = UnityObjectToClipPos(v.vertex);
o.pos.z -= 0.001 * o.pos.w; // 在裁剪空间微调深度
```

---

## 一句话总结

> **观察空间**：摄像机坐标系下的真实三维空间，适合做**几何计算、光照、法线变换**；距离和方向有真实物理意义。
>
> **裁剪空间**：投影已作用但未除以W，是**顶点着色器的必须输出格式**，适合做**屏幕对齐、深度控制、裁剪判断**；坐标已失去真实的距离意义。

## 变换矩阵速查（Unity HLSL）

| 需求 | 矩阵 |
|------|------|
| 模型空间 → 观察空间 | `UNITY_MATRIX_MV` |
| 观察空间 → 裁剪空间 | `UNITY_MATRIX_P` |
| 模型空间 → 裁剪空间 | `UNITY_MATRIX_MVP`（或 `UnityObjectToClipPos`） |
| 法线（模型→观察）   | `UNITY_MATRIX_IT_MV`（逆转置，防止缩放导致法线错误） |
| 世界空间 → 观察空间 | `UNITY_MATRIX_V` |
